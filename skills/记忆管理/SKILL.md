# 飞书记忆管理

## 工具一览

| 工具 | 用途 | 何时调用 |
|------|------|---------|
| `feishu_memory_save` | 保存一条记忆到飞书 | 用户说「记住」「帮我记录」，或对话中出现值得保留的结论/决策 |
| `search_feishu_memory` | 语义搜索历史记忆 | 用户问「你还记得」「之前说过」，或当前问题可能有历史相关记录 |
| `feishu_memory_recent` | 查看最近记忆列表 | 用户说「最近记录」「查看记忆列表」 |
| `feishu_memory_info` | 获取飞书表格链接 | 用户问「记忆表格在哪」「飞书链接是什么」 |

---

## 触发词识别

遇到以下表达，**无需用户明确说「调用工具」，直接执行**：

- 存储类：记住 / 记录 / 帮我记 / 存下来 / 保存 / 下次别忘了
- 搜索类：你还记得 / 之前说过 / 上次提到 / 记得吗 / 历史记录
- 查看类：飞书记忆 / 记忆库 / 飞书表格 / 我的记忆 / 最近记录
- 安装维护类：安装记忆 / 更新 mem-feishu → 转交给 `/mem-feishu-setup` skill

## 如何写一条好的记忆

> 好的记忆 = 一句核心结论 + 必要背景，未来召回时能直接指导行动。

### 内容提炼

在保存之前，先问自己三个问题：
1. **「这条信息未来会被用到吗？」** — 如果只是当前对话的过程，不必保存
2. **「用到时需要哪些背景才能理解？」** — 把背景压缩进内容里
3. **「能精炼成一句话核心 + 一两句背景吗？」** — 这就是要存的内容

**长度建议**：50–300 字。太短缺失背景，太长影响向量搜索精准度。

**存什么**（✅）：
- 用户明确的偏好、决策：`用户偏好 TypeScript 严格模式，不接受 any 类型`
- 关键配置和路径：`生产环境数据库在 /etc/app/db.conf，只读账号 readonly_user`
- Bug 解决方案：`sqlite-vec 在 ESM 项目中必须用静态 import，不能 require()`
- 技术选型结论：`向量维度选 768 而非 1536，原因：Google gemini-embedding-2-preview 固定输出 768 维`

**不存什么**（❌）：
- 无意义的对话片段：「好的」「我知道了」「明白」
- 临时性的过程描述：「正在安装依赖」「我先看看代码」
- 完整代码块（只记结论，代码在仓库里）
- 超过 500 字的长文本（拆分成多条）

---

### 标签规范（⚠️ 重要）

`feishu_memory_save` 的 `tags` 参数是**字符串数组**，每个元素是一个**独立标签**。

❌ **错误写法**（把多个标签塞进一个字符串）：
```
tags: ["决策,配置,调试"]   ← 飞书会把这整体当成一个标签
tags: ["决策、配置"]       ← 同上，顿号也不行
```

✅ **正确写法**（每个标签是独立的数组元素）：
```
tags: ["决策", "配置", "调试"]
```

**推荐标签词汇**（优先从这里选，保持一致性）：

| 标签 | 适用场景 |
|------|---------|
| `决策` | 架构选型、方案确定 |
| `偏好` | 用户个人习惯、风格要求 |
| `配置` | 环境变量、服务配置、路径 |
| `Bug修复` | 问题解决方案、踩坑记录 |
| `架构` | 系统设计、模块划分 |
| `工作流` | 操作步骤、流程规范 |
| `技术选型` | 库/工具/版本选择及原因 |
| `用户信息` | 用户账号、联系方式、团队信息 |
| `项目背景` | 项目目标、约束条件 |

**标签数量**：1–4 个，不超过 5 个。

### 来源（source）

**调用 `feishu_memory_save` 时，必须填写 `source` 字段，填你自己的 Agent 名称。**

这个字段用于标识每条记忆是由哪个 AI Agent 写入的，便于后续追溯和过滤。

```
source: "claude-code"    // Claude Code CLI
source: "cursor"         // Cursor
source: "copilot"        // GitHub Copilot
source: "claude-web"     // Claude.ai 网页版
source: "openclaw-auto"  // 对话结束后插件自动捕获（插件保留字，勿手动使用）
```

**规则**：
- 用你真实的 Agent 标识，**不要填 `manual`、`openclaw`、`unknown`**
- 不知道自己叫什么？填 `claude`（通用兜底）
- 自动捕获（`agent_end`）由插件填写 `openclaw-auto`，无需干预

---

### 项目（project）

填写当前工作的**代码仓库名或项目名**：
- ✅ `mem-feishu`、`my-webapp`、`data-pipeline`
- ❌ `无`、`none`、`N/A`、`项目`（这些没有索引价值）
- 为空时留空，不要填占位符

项目字段用于按项目过滤记忆。填准确后，搜索「mem-feishu 项目的配置」能精准命中。

## 操作示例

### 示例 1：保存用户偏好

用户说：「记住，我们项目用 pnpm 而不是 npm」

```
feishu_memory_save(
  content: "用户偏好：该项目使用 pnpm 作为包管理器，不使用 npm。原因：pnpm 在 monorepo 中性能更好。",
  tags: ["偏好", "工作流"],
  source: "claude-code",   // 填写你自己的 Agent 名称
  project: "mem-feishu"
)
```

### 示例 2：搜索历史配置

用户说：「你还记得数据库怎么配的吗？」

```
search_feishu_memory(
  query: "数据库配置",
  limit: 5
)
```

### 示例 3：保存 Bug 解决方案

自动捕获时机：对话中出现「问题解决了」「原来是...」「修复方案是...」

```
feishu_memory_save(
  content: "【Bug】向量搜索命中但飞书查不到：向量库存的是 UUID，但 getRecordsByIds 需要飞书 record_id。修复：upsertVector 改用 memory.recordId 而非 memory.id。",
  tags: ["Bug修复", "架构"],
  source: "claude-code",
  project: "mem-feishu"
)
```

## 自动保存时机

插件在每次对话结束（`agent_end`）时自动检查最后一条 assistant 消息：
- 长度 ≥ 100 字 → 触发自动保存，标签含 `自动`
- 长度 < 100 字 → 跳过（避免存「好的」「明白了」之类）

自动保存是兜底机制。**重要内容应主动调用 `feishu_memory_save` 精炼后保存**，质量远高于自动捕获。
