# mem-feishu-v2 讨论要点与待决策项 (v1.3)

Bill，针对你提出的 3 个新问题，我进行了深入的调研。特别是关于"扫码授权"的思路，非常有启发性！以下是详细的分析和最终方案：

## 1. 安装指引中的 Token 提取逻辑
**你的问题**：如果没获取到配置里的 apptoken，就要问用户之前是否建立过记忆库，提供 token 或者多维表格网址，我们需要让 agent 知道提取 token。

**解决方案**：
完全同意。我们会在 `install-guide.md` 中加入明确的对话分支逻辑：
1. Agent 会先问："您是第一次使用 mem-feishu-v2，还是已经在其他设备上创建过记忆库？"
2. 如果是第一次，Agent 引导用户提供 App ID/Secret，并调用 `setup` 工具自动建表。
3. 如果是老用户，Agent 会说："请提供您之前生成的 App Token，或者直接把飞书多维表格的网址发给我。"
4. 插件的 `setup` 工具会内置一个正则解析器，如果用户发来的是网址（如 `https://xxx.feishu.cn/base/bascnXXXXX`），工具会自动提取出 `bascnXXXXX` 并保存到配置中。

## 2. 飞书 OAuth 扫码授权（替代自建应用）
**你的问题**：如果模仿飞书自家的龙虾的做法，用用户自己的用户权限呢？比如配置过程就是 agent 给用户发个二维码让用户飞书扫码？这样省的自定义机器人了。

**深度调研结果**：
这个思路（OAuth 2.0 授权码流程）在技术上是可行的，而且确实能解决"多维表格所有权"的问题。但我仔细研究了飞书的 OAuth 机制后，发现它在**纯本地插件环境**下存在几个致命的体验问题：

1. **仍然需要 App ID**：飞书的 OAuth 扫码登录，必须由一个"飞书应用"发起。这意味着，要么我们（开发者）提供一个公共应用（需要飞书商店审核，且不能把 Secret 写死在开源代码里），要么用户还是得自己去创建一个自建应用。
2. **回调地址（Redirect URI）配置繁琐**：扫码后，飞书需要把授权码发给本地插件。这要求插件在本地启动一个临时 HTTP 服务器（如 `http://localhost:19280/callback`），并且**用户必须在飞书开发者后台手动把这个地址填入安全设置中**。这比填 App ID 还要麻烦。
3. **Token 过期问题**：扫码获取的 `refresh_token` 只有 30 天有效期。如果用户 30 天没打开 OpenClaw，记忆插件就会失效，必须重新扫码。

**最终建议**：
虽然扫码听起来很酷，但在本地插件场景下，它反而增加了用户的配置成本（必须配回调地址）和维护成本（30天过期）。
**我们还是坚持"自建应用 + 邮箱添加协作者"的方案**。用户只需要复制粘贴一次 ID 和 Secret，插件就会用用户的邮箱把多维表格的最高权限（Full Access）赋予用户。这是目前最稳定、配置成本最低的方案。

## 3. 定时同步任务（Cron Job）
**你的问题**：在安装的指令中尝试让 Agent 加入一个定时任务，每天同步向量数据库和飞书多维表格。能不能直接通过代码注入？

**深度调研结果**：
我查阅了 OpenClaw 最新的 Cron 架构文档。目前插件系统（Plugin SDK）还没有正式开放"代码直接注册 Cron"的 API（GitHub 上有相关的 RFC 讨论，但还没落地）。

**最佳解决方案（混合模式）**：
我们采用"Agent 创建 Cron + 插件后台兜底"的双重保障机制：

1. **Agent 创建 Cron（主同步）**：
   在 `install-guide.md` 中，我们会让 Agent 执行一段内置的 CLI 命令：
   ```bash
   openclaw cron add --name "mem-feishu-sync" --cron "0 3 * * *" --session isolated --message "请调用 mem_feishu_sync 工具，同步飞书多维表格和向量数据库的数据。" --announce
   ```
   这会在每天凌晨 3 点，让 Agent 在后台（isolated session）自动调用我们的同步工具。

2. **插件后台兜底（registerService）**：
   为了防止用户误删 Cron 任务，我们会在插件的 `index.ts` 初始化时，利用 `registerService` 启动一个轻量级的 `setInterval` 定时器（比如每 4 小时执行一次）。这个定时器完全由代码控制，不依赖 Agent，作为同步的兜底保障。

---

**总结**：
你的建议让我们的安装流程和同步机制变得非常健壮！
1. Token 提取：支持从网址自动解析。
2. 权限方案：维持自建应用+邮箱协作者（避开 OAuth 的坑）。
3. 同步机制：Agent Cron 任务 + 插件代码级定时器双重兜底。
